Add an ABS Census source for every NSW postcode

Produce a normalized, partitioned ABS Census profile (population, income, dwelling structure, tenure, rent and mortgage brackets, and the rest of the GCP profile) for every NSW postcode, on the same pull → extract → transform → manifest contract as nswgov and rentboard, so the downstream database project can join it onto sales and rental bond data by postcode.

Decision: build POA (postcode) only — it's the one geography that joins onto both nswgov and rentboard. SAL (suburb) is out of scope; see below.

Goal

One normalized row per NSW postcode (and, secondarily, per NSW suburb) carrying the same kind of stats shown on an ABS QuickStats page — population, age, income, dwelling type, tenure, rent/mortgage brackets.

Constraint

rentboard only has a postcode column, no suburb name. So postcode (ABS geography code POA) has to be the join key that actually works across both existing sources.

Recommendation

No suburb→LGA mapping step is needed. ABS already publishes Census data natively by postcode (POA) and by suburb/locality (SAL) — both are first-class ASGS geographies, not derived from LGA. Decided: build POA only for v1 since it's the key that actually joins both existing sources.

Why you don't need a suburb→LGA mapping

The QuickStats URL in the request (/quickstats/2021/LGA14000) happens to use an LGA (council area) code, but ABS publishes the identical style of Census profile at several independent geography levels — including POA (postal areas, i.e. postcodes) and SAL (suburbs and localities) — via Census DataPacks. LGA is just one geography among many; it is not an intermediate step you have to route postcodes/suburbs through.

Join-key strategy: POA vs SAL vs LGA

Decided, v1

POA (postcode)

The only geography both nswgov (postcode) and rentboard (postcode) can reliably join on. This is the whole v1 build.

Decided: out of scope

SAL (suburb)

Would only ever cover nswgov.locality, never rentboard (no suburb field). Since postcode already joins both sources, SAL adds a second pipeline surface for no join-coverage gain — not built.

Not needed

LGA (council)

Neither existing source has a council field, and ABS doesn't require routing through LGA to get postcode/suburb data. Skip unless a future consumer asks for it.

Data source: Census DataPacks (bulk CSV), not the QuickStats web pages

The QuickStats URL in the request renders one HTML page per geography — scraping ~640 postcode pages (or ~2,500 suburb pages) one at a time would be slow and fragile. ABS instead publishes the same underlying numbers as bulk ZIP/CSV downloads, pre-filterable to NSW, at Census > Find Census data > DataPacks: the General Community Profile (GCP), at POA geography, for NSW — the full table set, not a curated subset (see below).

Normalized Row Shape (POA)

One row per postcode, but wide — keeping the full GCP table set means every table contributes columns, so a row has hundreds of fields, not a dozen. Unlike nswgov/rentboard's hardcoded FINAL_COLUMNS, the column list here is derived at extract time from the metadata workbook (short code → label), because it's too large to hand-maintain and can shift slightly between Census releases.

Census Tables Pulled (full GCP set)

Decided: keep as much information as possible

The GCP DataPack ships ~60 tables (G01–G60+). Rather than curating down to a property-specific subset, v1 pulls every table the POA-level GCP DataPack provides and keeps it in the normalized output — the database project downstream can select/prune what it needs, but the collection layer shouldn't throw information away.

Implementation Plan

  • transform_abs
  • CLI + tests

1. Paths + core

Add abs_poa_dir, abs_sal_dir, abs_poa_manifest, abs_sal_manifest to core/paths.py. No new core mechanics needed — reuse atomic_write_csv and write_manifest.

2. pull_abs

Download the GCP DataPack ZIP for NSW at POA, plus the metadata/geography-label workbook, into data/raw/abs/ (gitignored, synced to Drive like the other sources' raw dirs); unzip. Idempotent: skip if the census year's raw files already exist. Keep the ZIP — it's static per census year, cheap to retain.

3. extract_abs

Iterate every table CSV present in the DataPack (not a hand-picked list), keep all short-code columns, melt to long form, decode short codes to labels via the metadata workbook.

4. transform_abs

Pivot wide to one row per postcode, attach the postcode label from the geography lookup, zero-pad postcode to match nswgov/rentboard, write partitioned output + refresh manifest.

5. CLI + tests

Add an abs subcommand (pull/extract/transform/update/manifest) mirroring nswgov's shape. Contract tests pin the curated column list and partition naming, using a small synthetic DataPack fixture rather than the real multi-MB download.

POA ≠ postcode boundary. ABS's Postal Area geography is a best-fit approximation built from mesh blocks, not the exact Australia Post boundary. A handful of postcodes (PO-box-only, very sparse) will have no Census data — treat as expected nulls, don't apply nswgov's "raise on zero rows" pattern here.
Cell perturbation. ABS applies small random noise to Census cell counts for privacy, especially in small geographies. Counts are indicative, not exact — worth a one-line note in the output's README so downstream consumers don't over-trust small-postcode figures.
Update cadence is fundamentally different. Census data changes every ~5 years (2021 → next release ~2027-28), not weekly. abs update is a rare, manual-trigger job, not something to wire into the same cron/schedule as nswgov/rentboard.
Wide output by design. Keeping the full GCP table set means a normalized row has hundreds of columns. That's an accepted tradeoff for maximizing retained information — downstream consumers select/prune what they need rather than the collector deciding for them.
