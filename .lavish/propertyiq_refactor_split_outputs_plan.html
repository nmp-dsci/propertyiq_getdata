Refactor propertyiq_getdata around partitioned source outputs

The collection repo should stop behaving like a downstream analytics store. Its job should be to fetch public source files, normalize each new period independently, and hand stable partitions plus manifests to the database project that does cleaning and serving.

Goal

Keep only the actively maintained NSW Valuer General and NSW Rental Bond collectors, and change final outputs from append-only monoliths to period-level files with a manifest.

Constraint

Downstream cleaning belongs in the database project, so this repo should preserve source-shaped normalized records, not denormalize or aggregate for product use.

Recommendation

Use CSV partitions first. It avoids new dependencies, works with existing Drive storage, and is easy for a DB loader to upsert by source and period.

Output Strategy Options

Recommended

CSV partitions + manifest

Lowest risk. Keeps dependencies stable and lets the DB project load changed periods only.

Later

Parquet partitions

Smaller and faster, but adds pyarrow and changes the storage contract more sharply.

Avoid

Keep giant CSV primary

Keeps current behavior, but every update and audit stays tied to scanning or appending huge files.

Compatibility risk: any downstream job that expects data/nswgov_df.csv and data/rentboard_df.csv needs either a migration or an explicit export-legacy step during transition.
Drive risk: rclone is installed locally but not configured here, so Drive path changes should be tested after credentials are available.
